<objective>
Add spatial layout awareness and auto-animate morphing to the visual stage. Panels intelligently position themselves based on what else is active, and consecutive same-module segments morph between states instead of destroying and recreating.

Purpose: The current system treats each panel show as independent — no awareness of other active panels, and same-module updates cause a full hide/show cycle. This plan makes multi-panel compositions feel spatial (panels dodge each other) and sequential data updates feel smooth (chart data morphs, map zooms transition).

Output: Spatial position resolver in VisualStageController, morph detection in StoryboardScheduler, CSS layout classes for split/overlay/pip, media-presenter position flow response.
</objective>

In storyboard-scheduler.js, modify `_fireSegment()` to detect morph opportunities:

1. **Track previous segment**: Add `this._lastFiredSegment = null` to constructor. After firing a segment, store it as `_lastFiredSegment`.

2. **Morph detection**: Before firing a segment, check if `_lastFiredSegment` exists and has the same `module` value (from segment.data). If both are chart_show (same chart panel), map_show (same map panel), or media (same media panel), this is a morph candidate.

3. **Morph flag**: When morph detected, add `segment.data._morph = true` to the message before dispatching to VisualStageController. This tells the stage NOT to hide/re-show but to update in place.

4. **Morph transition CSS class**: Add `_morphTransition = true` flag alongside `_morph` so the panel knows to use CSS transition for the data change (e.g., chart axes smoothly rescale).

In visual-stage.js, update `handleMessage()` and `_fadeIn()`:

1. **Morph path in handleMessage**: If `msg._morph === true` and the target panel is already active, skip the hide-oldest/queue logic entirely. Instead, call `panel.handleMessage(msg)` directly — the panel updates its content in place without a show/hide cycle.

2. **Chart morph**: ECharts panels already support `setOption()` with animation — calling handleMessage with new data will trigger ECharts' built-in morphing. No extra work needed for chart panels.

3. **Map morph**: Map panels receiving a morph message should call `flyTo()` instead of recreating. If the map panel has a `flyTo` method, call it; otherwise fall through to normal handleMessage.

4. **Media morph**: MediaPresenter already has crossfade between images. A morph flag means it should use the crossfade path (which it already does) rather than hide/show.
