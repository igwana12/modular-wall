Add `data_type` value "rss" to the docstring. Add optional field `items: list[dict[str, Any]] = Field(default_factory=list)` for list-style payloads (RSS entries, Twitter threads) alongside the existing `data` dict. This avoids overloading `data` for list vs dict semantics.

**2. Refactor DataPipeline to adapter registry (data_pipeline.py):**
Replace the hardcoded if/elif chain in `fetch()` with a registry pattern:

```python
# In __init__:
self._adapters: dict[str, Callable] = {}
self._register_defaults()

def _register_defaults(self):
    """Register the 4 built-in sources."""
    self.register_source("perplexity", self._fetch_perplexity)
    self.register_source("grok", self._fetch_grok)
    self.register_source("hyperliquid", self._fetch_hyperliquid)
    self.register_source("resourcerer", self._fetch_resourcerer)
    self.register_source("rss", self._fetch_rss)
    self.register_source("grok_twitter", self._fetch_grok)  # alias

def register_source(self, name: str, handler: Callable) -> None:
    """Register a new data source adapter."""
    self._adapters[name] = handler
```

Update `fetch()` to use `self._adapters.get(source)` instead of if/elif. Keep backward compat — existing 4 sources work identically.

Update `_detect_content_type` tool definition: add "rss" and "grok_twitter" to the source enum. Update the system prompt to explain: "rss" is for when user asks about news feeds, blog updates, subscriptions; "grok_twitter" is for Twitter/X specific trend/thread requests (vs general "grok" for research).
