**1. `firmware/oracle-engine/ws_client.h` and `ws_client.cpp` — ArduinoWebSockets client:**

Uses the ArduinoWebSockets library (add to README: install via Arduino Library Manager).

- `bool ws_connect(const char* host, int port, const char* path)`:
  - Create WebSocketsClient instance
  - Connect to ws://host:port/path
  - Register event handler for connected/disconnected/text/binary events
  - Return connection status
  - Reconnect logic: retry every 3 seconds if disconnected

- `void ws_loop()`:
  - Must be called in Arduino loop() — handles WebSocket events
  - Calls webSocket.loop()

- `bool ws_connected()`:
  - Return current connection state

- `void ws_send_config(const char* deity_id, const char* intent)`:
  - Send JSON: {"type": "config", "deity_id": deity_id, "intent": intent}
  - Use ArduinoJson for JSON serialization

- `void ws_send_audio(const uint8_t* pcm_data, size_t len)`:
  - Base64 encode the PCM data
  - Send JSON: {"type": "audio", "data": "<base64>"}
  - Use Arduino's base64 encoding (mbedtls_base64_encode on ESP32)

- `void ws_send_end_audio()`:
  - Send JSON: {"type": "end_audio"}

- Callback registration:
  - `void ws_on_status(void (*cb)(const char* state))` — state changes
  - `void ws_on_transcript(void (*cb)(const char* text))` — STT result
  - `void ws_on_audio(void (*cb)(const uint8_t* pcm, size_t len))` — TTS audio chunks (base64 decoded)
  - `void ws_on_done(void (*cb)())` — reading complete
  - `void ws_on_error(void (*cb)(const char* msg))` — error

- Internal message handler parses JSON messages and dispatches to registered callbacks
- For "audio" messages: base64 decode the data field and pass raw PCM bytes to the audio callback

**2. Update `oracle-engine.ino` — State machine:**

Disable LOOPBACK_TEST when WebSocket mode is active. Add a `#define WS_MODE` in config.h (default: enabled, replaces LOOPBACK_TEST).

State machine:
```
IDLE -> CONNECTING -> READY -> LISTENING -> PROCESSING -> SPEAKING -> READY
```

- `STATE_IDLE`: Initial state, attempting WiFi
- `STATE_CONNECTING`: WiFi connected, connecting WebSocket
- `STATE_READY`: WebSocket connected, waiting for PTT button press
- `STATE_LISTENING`: PTT held, capturing audio and streaming to backend
- `STATE_PROCESSING`: PTT released, end_audio sent, waiting for response
- `STATE_SPEAKING`: Receiving and playing TTS audio chunks

Implementation:
- On boot: connect WiFi -> connect WebSocket -> send config message with default deity ("apollo") and intent ("oracle guidance")
- PTT button (GPIO 0) handling:
  - Button pressed (LOW, with 50ms debounce): transition to LISTENING
  - While LISTENING: capture audio buffers via audio_capture(), send each via ws_send_audio()
  - Button released: send ws_send_end_audio(), transition to PROCESSING
- Audio callback: when "audio" messages arrive, feed decoded PCM to audio_playback()
  - Buffer incoming audio chunks into a playback ring buffer (8KB)
  - Play from ring buffer to avoid gaps between chunks
- On "done": transition back to READY
- On error or disconnect: transition to CONNECTING, attempt reconnect

**3. Latency instrumentation:**
- Record millis() timestamps at:
  - `t_ptt_release`: when PTT released (end of speech)
  - `t_transcript`: when transcript message received
  - `t_first_audio`: when first audio chunk received
  - `t_last_audio`: when done message received
- Print latency breakdown to Serial after each round-trip:
  ```
  === LATENCY REPORT ===
  STT:     {t_transcript - t_ptt_release} ms
  LLM+TTS: {t_first_audio - t_transcript} ms
  Total:   {t_last_audio - t_ptt_release} ms
  ======================
  ```
